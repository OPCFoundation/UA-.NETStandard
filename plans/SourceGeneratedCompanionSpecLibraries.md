# Plan: Source-Generated Companion-Spec Libraries (Machinery, Pumps)

> Status: **proposed** — work scoped for a follow-up PR.

## Why

The current `Applications/MinimalPumpServer` loads
`Opc.Ua.Machinery.NodeSet2.xml` and `Opc.Ua.Pumps.NodeSet2.xml` as
embedded resources at runtime. Companion-spec models are first-class
in this repo (DI is already a library — `Libraries/Opc.Ua.Di/`); the
same treatment should apply to Machinery and Pumps so consumer
applications get:

1. **AOT-safe deployment** — no XML parsing at startup; no
   `<EmbeddedResource>` payload baked into application binaries.
2. **Typed proxies** — source-generated `*TypeState` (server) and
   `*TypeClient` (client) classes for browse-/read-/method-call paths.
3. **Smaller assemblies** — model data lives in the library once, not
   in every consumer.
4. **Dependency tracking** — `Opc.Ua.Pumps` declares a project
   reference to `Opc.Ua.Machinery`, which declares a reference to
   `Opc.Ua.Di`. Versioning is enforced by NuGet/MSBuild.

## What ships

| New library | ModelDesign source | Server / client packages |
|-------------|--------------------|--------------------------|
| `Libraries/Opc.Ua.Machinery/` | `OpcUaMachineryModel.xml` (+ CSV) | model-only |
| `Libraries/Opc.Ua.Pumps/` | `OpcUaPumpsModel.xml` (+ CSV) | model-only |

Mirrors the layout of `Libraries/Opc.Ua.Di/`:

```
Libraries/Opc.Ua.Machinery/
  Design/
    OpcUaMachineryModel.csv
    OpcUaMachineryModel.xml
  Opc.Ua.Machinery.csproj
```

Server / client trios (`Opc.Ua.Machinery.Server`,
`Opc.Ua.Machinery.Client`) are optional — add them only when a
dedicated node-manager surface or client helper class is justified.

## Source-generator gaps to fix first

The model libraries cannot be source-generated until the following
are resolved in `Tools/Opc.Ua.SourceGeneration.Core/`:

### 1. Cross-namespace type references

When Pumps' ModelDesign declares a TypeDefinition in the DI namespace,
the generator emits `global::DI.PumpDeviceTypeState` based on the
`XmlPrefix` attribute. Our DI library uses C# namespace `Opc.Ua.Di`
(the `Prefix` attribute), so the emitted symbol does not resolve.

**Fix**: the generator must consume the ModelDesign `<Namespaces>`
table and use the C# `Prefix` of each referenced namespace, not the
XML prefix. Map by `Uri`. When the prefix is unknown (e.g. a runtime
NodeSet2 import without a corresponding ModelDesign mapping), fall
back to the XML prefix as today.

### 2. Re-verification of G9 / G10 / G11 in Machinery + Pumps

G9 (cross-namespace BrowseNames), G10 (OptionSet defaults), and G11
(placeholder children) were fixed against DI. Machinery + Pumps may
expose additional corner cases (e.g. nested OptionSets, deeper
placeholder chains) that surface only when their generators run.

**Fix**: add `Tests/Opc.Ua.SourceGeneration.Tests/` cases for each
gap reproducer against a stripped-down Machinery / Pumps fixture.

## Pre-requisites

- **ModelDesign XMLs** for Machinery and Pumps. Currently only the
  NodeSet2 XMLs ship with the OPC UA spec packages. Conversion paths:
  1. **OPC UA ModelCompiler** (`OpcUaCompiler.exe`) accepts NodeSet2
     input and emits ModelDesign XML; rerun against the upstream
     `OPCFoundation/UA-Nodeset/Pumps` and `Machinery` directories.
  2. **Author by hand** — for small surfaces this is feasible; for
     Pumps (6.4 MB NodeSet2) it is not.
- **CSV definition files** generated by the same tool (NodeIds.csv,
  BrowseNames.csv) — used by the generator for stable id reservation.

## Library layout

Each library follows the same template as `Libraries/Opc.Ua.Di/`:

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <AssemblyName>$(AssemblyPrefix).Machinery</AssemblyName>
    <TargetFrameworks>$(LibTargetFrameworks)</TargetFrameworks>
    <PackageId>$(PackagePrefix).Opc.Ua.Machinery</PackageId>
    <RootNamespace>Opc.Ua.Machinery</RootNamespace>
    <Description>OPC UA Machinery (OPC 40001) information model</Description>
    <IsAotCompatible Condition="…net10.0…">true</IsAotCompatible>
  </PropertyGroup>
  <ItemGroup>
    <ProjectReference Include="..\..\Stack\Opc.Ua.Core\Opc.Ua.Core.csproj" />
    <ProjectReference Include="..\Opc.Ua.Di\Opc.Ua.Di.csproj" />
  </ItemGroup>
  <ItemGroup>
    <ProjectReference Include="..\..\Tools\Opc.Ua.SourceGeneration\…">
      <OutputItemType>Analyzer</OutputItemType>
      <ReferenceOutputAssembly>false</ReferenceOutputAssembly>
    </ProjectReference>
  </ItemGroup>
  <Import Project="..\..\Tools\Opc.Ua.SourceGeneration\OPCFoundation.Opc.Ua.SourceGeneration.props" />
  <ItemGroup>
    <AdditionalFiles Include="Design\OpcUaMachineryModel.csv" />
    <AdditionalFiles Include="Design\OpcUaMachineryModel.xml" />
  </ItemGroup>
</Project>
```

Pumps adds a project reference to `Opc.Ua.Machinery`.

## Consumer impact

After conversion, `Applications/MinimalPumpServer` simplifies to:

```csharp
NodeStateCollection nodes = new ModelLoaderBuilder()
    .AddModel((coll, ctx) => coll.AddOpcUaDi(ctx))
    .AddModel((coll, ctx) => coll.AddOpcUaMachinery(ctx))
    .AddModel((coll, ctx) => coll.AddOpcUaPumps(ctx))
    .Build(new NodeStateCollection(), context);
```

The `Model/` directory and the `<EmbeddedResource>` items in
`MinimalPumpServer.csproj` go away. Typed proxies (e.g.
`Opc.Ua.Pumps.PumpDeviceTypeState`) become available so fluent
wiring no longer relies on browse-path strings.

## Risk / open questions

- **Machinery 1.04+ optional dependency on IA** — older Pumps depends
  on Machinery 1.01. Need to pin the source NodeSet2 version. The
  current sample is built against the
  `Pumps-1.0.0-2021-04-19` tag in `OPCFoundation/UA-Nodeset`.
- **Code-gen output volume** — Pumps has ~10k nodes; the generated
  C# file will be large. Measure assembly size before / after.
- **Naming collisions** with WoT-Con / GDS proxies that already exist
  in the repo are unlikely (different namespace prefixes) but worth
  verifying.

## Out of scope for this plan

- Onboarding facet (already split into a separate
  `Opc.Ua.Onboarding.*` future work item — see
  `Docs/Opc.Ua.Di/ComplianceMatrix.md`).
- Any companion spec not currently used by samples (Robotics, IA,
  ProcessAutomation, etc.). Each ships its own follow-up PR using
  this same template.
